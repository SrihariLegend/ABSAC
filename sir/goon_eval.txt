### 1. Should `complexity_penalty` exist on `TransformationScore`?

The Cost Model layer is explicitly marked as "Not started". Adding structures like `TransformationScore` or fields like `complexity_penalty` now is premature. You should defer this design until the cost model layer is active.

### 2. Should `Proof` be in the `CostModel::evaluate` signature?
**Recommendation:** **No.**

**Reasoning:**
- **Violates Feed-Forward Pipeline:** The architecture enforces that *"Each layer consumes only the knowledge of the immediately preceding layer. No layer reads upward or across."* 
- **Pipeline Order:** Verification → Rewrite → Cost → Selection.
- The `Cost` layer immediately follows `Rewrite`. Therefore, it should only consume the outputs of `Rewrite` (e.g., `Mutations` or mutated graphs), not the outputs of `Verification` (`Proofs`). Passing `Proof` directly into `CostModel::evaluate` would bypass the rewrite layer. 
- **Semantic mismatch:** A `Proof` establishes semantic equivalence, which has no bearing on the computational cost of the mutation. The cost model only needs to evaluate the performance characteristics of the transformed graph.
Furthermore, a `Proof` establishes semantic equivalence—it has no bearing on execution cost, making it semantically irrelevant to the cost model.
